    Book Progress Dashboard — Specification

    Status: Draft v1 Audience: Engineers extending the dashboard, reviewers evaluating fitness for purpose, and authors of the acceptance-test-plan.md. Implementation under specification: docs/dashboard/ (React 18 + Babel-standalone single-page app).

    This is a specification-driven design (SDD) document. Every behavior the dashboard must exhibit is captured here as a numbered requirement that can be traced to a test in the acceptance plan. If the implementation and this document disagree, fix the one that is wrong — they are intended to stay in lockstep.

    1. Purpose and Goals

    The dashboard exists to help authors of intelligent textbooks answer four questions, fast:

    1. Where am I? What fraction of the book is complete, broken down by build step.
    2. What's next? Which task should I run right now, and how much will it cost?
    3. Will I finish in budget? How does pending work compare against my plan's 5-hour token window?
    4. Which book am I working on? Switch between books without changing code.

    The goals stated in the home page (index.md) drive the requirements below: track progress, view past token usage, predict tokens needed, surface burn-down charts, list required and optional tasks, recommend the next task, and remind authors that not every step needs the most expensive model.

    2. Stakeholders and User Roles

    Role Description Primary Concerns
    Author The person writing the textbook. Drives most interaction. Progress, what's next, what it costs.
    Editor / Reviewer Reviews drafts, transitions sections from drafted → reviewed → published. Section drill-down, child item lists, status counts.
    Project Lead Tracks multiple books at once. Book selector, comparing scope/progress between books.
    Onboarding Reader New to intelligent-textbook generation. The DAG of build steps, the description of each step.

    The dashboard MUST serve all four roles from the same UI without role-specific modes or logins.

    3. Glossary

    Term Definition
    Step / Section A unit of work in the textbook build process (e.g. Course Description, Learning Graph, Quiz). Defined in workflow-steps.json. The two terms are interchangeable; the JSON uses step, the UI uses section.
    Status One of not-started, drafted, reviewed, published. See §6.1.
    Aggregate section A section with a children array (e.g. Chapters, Microsims). Its progress and status roll up from its children.
    Leaf section A section without children. Its progress is determined entirely by its own status.
    Book status file A per-book JSON in docs/book-data/<slug>.json describing one book's project metadata + per-section status.
    Manifest docs/book-data/books.json — the list of books the selector renders. Auto-generated by mkdocs-hooks/sync_books_manifest.py.
    Slug A book's filename stem (e.g. algebra-book-status for algebra-book-status.json). Used as the ?book= query parameter.
    Window The rolling 5-hour token allowance on Claude Pro / Max plans. Sized by PROJECT.sessionBudget[plan].
    Burn A bulk run of every pending task, used to spend remaining window tokens before they expire.

    The dashboard runs entirely client-side. There is no backend. Data is read from three JSON files at startup, merged into the legacy globals (window.SECTIONS, window.PROJECT, window.STATUS_WEIGHT, window.STATUS_META), and React mounts only after window.DATA_READY resolves.

    5. Data Model

    5.1 workflow-steps.json schema

    Authoritative shared definition; one file for all books.

    {
  "constants": {
    "statusWeight": { "<status>": <0..1> },
    "statusMeta":   { "<status>": { "label": str, "dot": cssVar, "fg": cssVar, "bg": cssVar } }
  },
  "steps": [
    {
      "id":              "kebab-case-id",
      "name":            "Display name",
      "short":           "Short label for narrow space",
      "description":     "1-3 sentence description",
      "dependsOn":       ["id-of-prerequisite-step", ...],
      "childIdPrefix":   "ch" | "sim" | <absent for leaf steps>
    }
  ]
}

    Constraints (CON-WS-*): - CON-WS-1: Every id MUST be unique within steps. - CON-WS-2: Every entry in dependsOn MUST refer to another step's id in the same file. - CON-WS-3: The set of dependsOn edges MUST form a DAG (no cycles). - CON-WS-4: statusWeight MUST contain exactly the four keys not-started, drafted, reviewed, published, with values in [0,1] monotonically non-decreasing in that order. - CON-WS-5: statusMeta MUST contain entries for the same four keys. - CON-WS-6: Steps with childIdPrefix are aggregate; steps without are leaf. The book status file MUST match this shape (see CON-BS-3).

    5.2 Book status file schema

    One file per book, at docs/book-data/<slug>.json.

    {
  "project": {
    "title":           "Display title",
    "subtitle":        "Free-form subtitle",
    "author":          "Author or You",
    "costPerMTokens":  <number>,
    "sessionBudget":   { "pro": <int>, "max": <int> }
  },
  "sections": {
    "<step-id>": {
      // leaf shape:
      "status":       "<status>",
      "tokensTarget": <int>,
      "tokensDone":   <int>
    },
    "<aggregate-step-id>": {
      // aggregate shape:
      "countLabel": "12 chapters",
      "children": [
        { "n": <int>, "title": str, "status": "<status>", "tokens": <int> }
      ]
    }
  }
}

    Constraints (CON-BS-*): - CON-BS-1: Every <step-id> key in sections MUST be the id of a step in workflow-steps.json. Unknown ids are ignored by the loader. - CON-BS-2: Steps that exist in workflow but are missing from sections default to status: "not-started", tokensTarget: 0, tokensDone: 0 (see DM-2). - CON-BS-3: A section entry's shape MUST match its workflow step's kind: leaf step → leaf shape, aggregate step → aggregate shape. - CON-BS-4: tokensDone <= tokensTarget SHOULD hold. The UI does not crash if violated, but Math.round((done/target)*100) may exceed 100. - CON-BS-5: Each child's n SHOULD be unique within its parent (it becomes part of the synthesized DOM id). Duplicate n values cause React key collisions.

    5.3 books.json manifest schema

    [
  { "slug": "<filename-stem>", "title": "Display title from project.title" }
]

    This file is regenerated by mkdocs-hooks/sync_books_manifest.py on every build. Hand edits will be overwritten.

    5.4 Data merge — DM-* requirements

    DM-1 (Loader globals): data.js MUST set the following globals before resolving window.DATA_READY:

    • window.WORKFLOW — the entire workflow JSON
    • window.PROJECT — <book>.project
    • window.STATUS_WEIGHT — workflow.constants.statusWeight
    • window.STATUS_META — workflow.constants.statusMeta
    • window.SECTIONS — the merged array (see DM-2)
    • window.BOOKS — the manifest array
    • window.CURRENT_BOOK — the slug currently loaded

    DM-2 (Merged section shape): For each step in workflow.steps, SECTIONS MUST contain a corresponding entry built as follows:

    • Always: { id, name, short, description } from the workflow step.
    • If step.childIdPrefix is present:
    • countLabel ← book status countLabel (or undefined)
    • children ← book status children, each augmented with id: \${step.childIdPrefix}-${c.n}``
    • Else (leaf):
    • status ← book status status or "not-started"
    • tokensTarget ← book status tokensTarget or 0
    • tokensDone ← book status tokensDone or 0

    DM-3 (Section order): SECTIONS MUST preserve the order of workflow.steps. The progress bar, overview grid, and next-task prioritization all depend on this order.

    DM-4 (Selected book): The slug to load comes from ?book=<slug> in the page URL; if absent, the loader defaults to gen-ai-book-status. The default MUST be a slug that exists in the manifest of any committed deployment.

    6. Functional Requirements

    6.1 Status Lifecycle (FR-STATUS-*)

    The four statuses form an ordered progression: not-started → drafted → reviewed → published.

                       ┌────────────┐
                   │ not-started│
                   └──────┬─────┘
                          │ author writes draft
                          ▼
                   ┌────────────┐
                   │  drafted   │
                   └──────┬─────┘
                          │ reviewer signs off
                          ▼
                   ┌────────────┐
                   │  reviewed  │
                   └──────┬─────┘
                          │ author marks complete
                          ▼
                   ┌────────────┐
                   │ published  │
                   └────────────┘
    • FR-STATUS-1: Each leaf section and each child item has exactly one status from this set.
    • FR-STATUS-2: An aggregate section's rollup status is computed from its children per helpers.jsx sectionRollupStatus:
    • All children published → published
    • All children not-started → not-started
    • All children reviewed (none not-started, none drafted, none published) → reviewed
    • Otherwise → drafted (interpreted as "in progress")
    • FR-STATUS-3: Each status is rendered with a colored dot whose CSS variable comes from STATUS_META[status].dot. The four colors MUST be visually distinct. Current palette: gray-300, amber-500, blue-500, green-500.

    6.2 Book Loading and Selection (FR-LOAD-*)

    • FR-LOAD-1: On page load, the dashboard MUST fetch workflow-steps.json, ../book-data/books.json, and ../book-data/<slug>.json in parallel and only mount React after all three resolve.
    • FR-LOAD-2: The page URL accepts a ?book=<slug> query parameter. The slug names a status file in book-data/ without the .json extension.
    • FR-LOAD-3: If ?book= is omitted, the default slug gen-ai-book-status MUST be used.
    • FR-LOAD-4: A <select> book-selector MUST render in the topbar listing every entry in the manifest, with the currently loaded book preselected.
    • FR-LOAD-5: Choosing a different book in the selector MUST navigate to the same URL with ?book=<new-slug> and reload, re-bootstrapping the dashboard with the new book's data.
    • FR-LOAD-6: When the manifest contains fewer than 2 entries, the selector MUST hide itself.

    6.3 Header — Title and Overall Progress (FR-HEADER-*)

    • FR-HEADER-1: The book's project.title MUST appear as the page H1.
    • FR-HEADER-2: The book's project.subtitle MUST appear directly below as a smaller line.
    • FR-HEADER-3: A large numeric overall-percent readout MUST display: 
      round(  Σ sectionDoneTokens(s)  /  Σ sectionTargetTokens(s)  * 100 )
      computed across every s in SECTIONS.
    • FR-HEADER-4: The denominator above MUST be > 0 for any reasonable book; if a book has zero total target tokens the readout MAY render NaN% (the UI does not need to special-case this).

    6.4 Progress Bar (FR-PROG-*)

    A horizontal segmented bar where each segment is one section.

    • FR-PROG-1: Segment widths MUST be proportional to sectionTargetTokens(s) / Σ sectionTargetTokens. Larger sections get wider segments.
    • FR-PROG-2: Each segment's fill MUST be Math.round(sectionProgress(s) * 100)%, where sectionProgress is defined in helpers.jsx.
    • FR-PROG-3: Each segment's color comes from sectionRollupStatus(s) per FR-STATUS-2 / FR-STATUS-3.
    • FR-PROG-4: Clicking a segment MUST select that section (open the detail panel). Clicking the already-selected segment MUST deselect.
    • FR-PROG-5: A row of segment labels (using s.short) MUST appear directly below the bar with the same widths and selection behavior.
    • FR-PROG-6: A legend MUST list all four statuses with their colored dots.

    6.5 Summary Cards (FR-SUMMARY-*)

    Four cards below the progress bar:

    • FR-SUMMARY-1 (Overall): Card 1 shows overall percent + formatTokens(totalDone) of formatTokens(totalTarget).
    • FR-SUMMARY-2 (Total budget): Card 2 shows the total target token count and its dollar cost at PROJECT.costPerMTokens. Hidden if "Show token estimates" is off (see FR-TWEAK-1).
    • FR-SUMMARY-3 (Time): Card 3 shows ~ceil(remaining / 80_000) days as a rough completion estimate.
    • FR-SUMMARY-4 (Sessions): Card 4 shows ceil(remaining / sessionBudget[plan]) 5-hour sessions left, with an inline Pro/Max toggle that updates plan for the whole dashboard.

    6.6 Overview Grid and Detail Panel (FR-DETAIL-*)

    • FR-DETAIL-1: When no section is selected, the dashboard MUST render an Overview Grid — one tile per section — with name, optional countLabel, percent fill, and a small bar.
    • FR-DETAIL-2: Clicking a tile (or a progress segment / segment label) MUST select that section.
    • FR-DETAIL-3: When a section is selected, the dashboard MUST render the Detail Panel with:
    • Breadcrumb back-link returning to the overview
    • Section name + countLabel if aggregate
    • Status pill (rollup status for aggregate)
    • Description
    • Action row: Generate, Preview, Regenerate, Mark complete, Notes (these are visual placeholders in v1 — see §10)
    • Stats sidebar: percent, tokens estimated, tokens generated, estimated cost, "last updated"
    • FR-DETAIL-4: For aggregate sections, the Detail Panel MUST show:
    • Status counts row (Published / Reviewed / Drafted / To do)
    • "{N} items" with an expand toggle
    • When expanded: numbered child rows with title, token count, and status pill

    6.7 Run Next Task (FR-NEXT-*)

    • FR-NEXT-1: The dashboard MUST surface a "Run next task" card prominently (between the summary strip and the detail shell).
    • FR-NEXT-2: The recommended task is selected by pickNextTask() in next-task.jsx, which prioritizes:
    • drafted items first (need review / completion polish)
    • not-started items next
    • Within a tier, the first item encountered in section order (DM-3)
    • published items are never recommended
    • FR-NEXT-3: If every section and every child is published, the card MUST show "All tasks complete 🎉" instead of being clickable.
    • FR-NEXT-4: Clicking the card opens a modal with: status pill, full description, estimated clock time, tokens for this task, window-remaining-after, dollar cost, and a mini-bar visualizing this task's chunk of the session window.
    • FR-NEXT-5: The modal MUST close on click outside, on the close button, and on Escape.
    • FR-NEXT-6: The "Run task" button in the modal is a placeholder action in v1 — clicking it dismisses the modal.

    6.8 Burn Tokens (FR-BURN-*)

    • FR-BURN-1: A "Got tokens to burn?" rail MUST appear below the detail shell with a Burn Tokens button.
    • FR-BURN-2: The button opens a modal that lists every pending (non-published) item from SECTIONS, with checkboxes (all preselected by default).
    • FR-BURN-3: The modal header MUST display: window-resets-in clock (h:mm:ss, decreasing), tokens left in window, and selected/burning total.
    • FR-BURN-4: The user MUST be able to choose parallel or sequential execution mode and the chosen mode persists for the duration of the modal session.
    • FR-BURN-5: When selected total > tokens-left-in-window, the modal MUST show an over-budget warning citing the overage in tokens.
    • FR-BURN-6: "Burn Now" runs an animation: in parallel mode every selected task starts at t=0 and completes after a randomized 6–14s; in sequential mode tasks are chained, each taking 4–8s.
    • FR-BURN-7: During burn, checkboxes and mode-radios MUST be disabled.
    • FR-BURN-8: When all tasks reach progress === 1, the modal MUST switch to "done" state with a "Run again" button.
    • FR-BURN-9: The window-resets-in clock turns urgent (red) when remaining time < 30 minutes.
    • FR-BURN-10: The modal MUST close on click outside, on close button, and on Escape.

    6.9 Tweaks Panel (FR-TWEAK-*)

    • FR-TWEAK-1: A "Show token estimates" toggle hides every UI element with the tokens-only CSS class when off. It defaults from window.TWEAK_DEFAULTS.showTokens (set in book-dashboard.html).
    • FR-TWEAK-2: A "Density" radio toggles between comfy and compact, written to document.documentElement.dataset.density. CSS reads this to adjust spacing.
    • FR-TWEAK-3: A "Plan" radio toggles between pro and max. The same value is read by §6.5 (sessions left) and §6.7 (window remaining).
    • FR-TWEAK-4: Tweak changes MUST take effect immediately without page reload.
    • FR-TWEAK-5: The tweaks panel itself opens via the host-mode protocol (__activate_edit_mode postMessage) and is irrelevant to standalone use.

    6.10 URL State (FR-URL-*)

    • FR-URL-1: The currently selected book is encoded as ?book=<slug> and is the only piece of state in the URL.
    • FR-URL-2: Other UI state (selected section, expanded child list, density, plan) is in-memory only and resets on reload.
    • FR-URL-3: Bookmarking ?book=<slug> MUST always restore the same book.

    6.11 Build Pipeline (FR-BUILD-*)

    • FR-BUILD-1: mkdocs-hooks/sync_books_manifest.py MUST run on mkdocs build and on every mkdocs serve rebuild.
    • FR-BUILD-2: The hook MUST glob docs/book-data/*.json, exclude books.json, and write a manifest sorted by filename.
    • FR-BUILD-3: Each manifest entry's title MUST be the project.title from that book's status file; if missing, the filename stem is used as a fallback.
    • FR-BUILD-4: The hook MUST be idempotent: a run that produces identical content MUST NOT modify the file (avoiding spurious git diffs).

    7. Non-Functional Requirements

    ID Requirement
    NFR-PERF-1 The dashboard MUST be interactive within 2 seconds on a modern laptop with all three JSON fetches served from mkdocs serve on localhost.
    NFR-PERF-2 A book with up to 20 sections, each with up to 100 children, MUST render without perceptible lag.
    NFR-A11Y-1 The book selector MUST be a native <select> (keyboard accessible).
    NFR-A11Y-2 The next-task and burn modals MUST close on Escape (FR-NEXT-5, FR-BURN-10).
    NFR-A11Y-3 Modal overlays MUST have role="dialog" and aria-modal="true".
    NFR-PORT-1 The dashboard MUST work in current Chrome, Firefox, and Safari. No IE11.
    NFR-PORT-2 The dashboard MUST work behind any static file server (e.g. mkdocs serve, python -m http.server). It MAY fail under file:// due to fetch/CORS restrictions; this is acceptable.
    NFR-DEP-1 All third-party JS dependencies are loaded from public CDNs with SRI hashes (see book-dashboard.html).
    NFR-OBS-1 The MkDocs hook MUST log a single info-level message when it regenerates the manifest, including the book count.

    8. Cost / Budget Model

    • CB-1: Dollar cost is computed as (tokens / 1_000_000) * PROJECT.costPerMTokens. Currency is USD.
    • CB-2: A 5-hour session spends PROJECT.sessionBudget[plan] output tokens.
    • CB-3: Sessions-left = ceil( (Σ tokensTarget − Σ tokensDone) / sessionBudget[plan] ).
    • CB-4: "Time to completion" uses 80,000 tokens / day as a working pace. This is heuristic; treat it as guidance, not a guarantee.
    • CB-5: The Burn modal hard-codes "220K tokens already used" as a session-window prelude. This is illustrative; future versions SHOULD source it from real session telemetry.

    9. Constraints

    • C-1: No backend. All state is either in-memory or in static JSON files.
    • C-2: No bundler step. JSX is transpiled in the browser by @babel/standalone.
    • C-3: All JSON files live under docs/ so MkDocs serves them as static assets at predictable URLs.
    • C-4: The dashboard supports exactly one book at a time per page load. Multi-book overlays are out of scope.
    • C-5: "Last updated" timestamps in the Detail Panel are placeholder text in v1.
    • C-6: No authentication, no multi-user editing, no real-time collaboration. The dashboard is a read-only view of static data.

    10. Non-Goals (Explicit Exclusions)

    The following are intentionally not part of v1 and SHOULD NOT be built without an updated specification:

    • Editing book data inline (status changes, token re-estimation). Status JSONs are edited by hand or by upstream tooling.
    • Authenticated user sessions.
    • Pulling real LLM session usage from any cloud provider's API.
    • Mobile-first responsive layout. The dashboard targets wide laptop / desktop screens.
    • Internationalization. UI strings are English-only.
    • Accessibility audit beyond keyboard support and modal semantics. WCAG AA conformance is a future goal.
    • The Detail Panel action buttons (Generate, Preview, Regenerate, Mark complete, Notes) are placeholders. Wiring them to actual model invocations is out of scope for v1.
    • The Run-task button in the Run Next Task modal is also a placeholder.
